# Hyper-V 仮想マシン 一括デプロイ実践ノート

Windows Server 2025 を Hyper-V 上で複数台デプロイする際の
実践記録です。

---

## 背景

授業で Windows Server を8台手動インストールする作業が発生しました。

4台目で手動作業の非効率さを実感し、PowerShell による自動化を検討しました。

試行錯誤の末、安定動作する一括デプロイスクリプトを完成させました。

---

## 構成概要
```
母片 VM（VM01）
└── 手動で OS インストール・初期設定済みのクリーン環境
    │
    ├── winser02
    ├── winser03
    ├── winser04
    ├── winser05  ← スクリプトで自動デプロイ
    ├── winser06
    ├── winser07
    └── winser08
```

---

## 技術的な課題と解決策

### 1. 母片（マスター VM）について

最初の1台は手動で構築する必要があります。

理由：OS インストール・パッチ適用・必要ツールの導入が完了した
クリーンな環境が複製の基準となるためです。

Windows Server 2025 は複製後の起動時にコンピューター名を
自動でランダム化するため、sysprep は不要であることを確認しました。

### 2. TPM 設定がスクリプトで失敗した原因

Windows Server 2025 では TPM に対してホスト側の暗号化設定が必要です。
```
失敗の原因
└── New-VMMachineKeyProtector は
    ホストが「ガーディアンモード」に
    初期化されていないと実行できない

GUI（Hyper-V マネージャー）
└── 「適用」ボタンクリック時に
    権限の関連付けを自動処理する

PowerShell
└── 実行順序が1つでもずれると
    権限エラーで拒否される
```

### 3. 設計判断：安定性優先

TPM 設定をスクリプトから除外しました。

理由：TPM の権限エラーによってスクリプト全体が停止し、
その後の「コンピューター名変更」「シャットダウントラッカー無効化」
が実行されなくなるリスクを避けるためです。
```
スクリプトで自動化した処理
├── VHD（仮想ハードディスク）の複製
├── VM の規格設定（CPU・メモリ）
├── コンピューター名の変更
└── シャットダウンイベントトラッカーの無効化

手動で対応した処理
└── TPM の有効化（GUI から統一設定）
```

この判断により、スクリプトの安定性と保守性を確保しました。

## トラブルシューティング記録

### コンピューター名を誤って変更した場合

スクリプトのテスト中に管理ホストのコンピューター名を
誤って変更してしまいました。

以下のコマンドで元の名前に戻しました：
```powershell
Rename-Computer -NewName "NEW-COMPUTER-NAME" -Restart -Force
```

教訓：スクリプトのテストは対象 VM を明確に確認してから実行する。

## 学んだこと

手動作業の非効率さを「自動化で解決する」という発想と、
自動化が困難な部分は「手動と組み合わせる」という判断が
実務では重要だと実感しました。
